Creative Component: AI Co-Designer System Blueprint (Thorough Update)
Project: simplechatbot
Date: 2026-04-22

Document goal
This file is the master explanation of what the software is, what it does today, and how it supports the core research objective:

Establishing design-style communication with AI through multimodal, human-led collaboration.

In simple terms:
- The system is not just a chatbot.
- It is a design communication environment.
- It helps the designer and AI co-construct ideas using text, voice, visuals, and evolving project context.

==================================================
1) Core Product Positioning
==================================================

Primary purpose
- Help a designer communicate intent, style, constraints, and decisions to AI across a full design process.

Design communication focus
- Ideation: broad exploration and framing
- Conceptualization: developing and critiquing direction
- Execution: concrete implementation and refinement guidance

What makes this a creative component
- Phase-adaptive prompting (mode-specific behavior)
- Multimodal inputs (voice, image, screen, camera, whiteboard artifacts)
- Persistent project memory (project info + chat continuity)
- Human-in-the-loop controls (user chooses mode, agents, inputs, and direction)

==================================================
2) Critical Clarification of Vision Inputs
==================================================

Live camera feed (prototype role)
- The live camera feed is a prototyping communication channel.
- Its purpose is to quickly show physical sketches, rough mockups, material samples, hand gestures, and in-progress prototypes to the AI.
- It is intended to increase communication bandwidth during design conversations, not to function as surveillance or passive monitoring.

Live screen capture (design artifact role)
- The live screen input is meant for sharing design artifacts such as:
  - storyboards
  - journey maps
  - interface flows
  - references and moodboards
  - CAD or layout snapshots
- It provides contextual evidence so the AI can critique, synthesize, and align outputs with what the designer is actually seeing.

Why this matters
- Camera supports communication from physical-world prototype to AI.
- Screen supports communication from digital design process to AI.
- Together, they create a richer "design style communication" loop than text alone.

3) Updated Feature Inventory (Current Implementation)
==================================================

A. Interaction and input channels
- Text chat input and threaded conversation history
- Voice dictation and voice interaction loop
- Image upload, paste, and drag/drop into chat
- Vision mode with selectable source:
  - screen capture
  - live camera feed
- Whiteboard as an interactive workspace for images/text/artifacts

B. AI behavior and orchestration
- OpenRouter-based chat completion path
- Streaming response path with non-stream fallback
- Phase mode logic from modes.conf:
  - ideation
  - conceptualization
  - execution
- Specialist agent prompt overlays:
  - brand master
  - design researcher
  - ergonomics agent
  - CMFT expert

C. Visual reasoning and generation
- Google Cloud Vision pre-analysis of visual inputs
- Runware image generation via WebSocket
- Text-to-image and image-to-image support
- Seed/reference image handling for iterative remix workflows

D. Project continuity and memory
- Multi-project management
- Multi-chat threads per project
- Product info extraction, merge, and auto-fill syncing
- Persistent local state to preserve design context across sessions

E. Output and transformation tools
- TTS output via kokoro-js and local ONNX model assets
- Whiteboard actions:
  - append selected images to chat
  - analyze selected images
  - remix selected images
  - request technical feasibility

F. Interface and runtime modes
- Full interface (whiteboard + chat)
- Bubble mode (compact always-on-top chat overlay)
- Electron desktop runtime with global quick-toggle shortcut

Backend and APIs
